# 费曼导师：上下文工程原则与写作标准

> 这份文件记录了和用户讨论费曼导师（`~/.claude/skills/feynman-tutor/`）重构时沉淀出的原则、标准、和用户的明确偏好。
>
> **使用场景**：每当用户想继续优化费曼导师时，先读这份 reference，再开始讨论。不要重新发明已经对齐过的设计选择——这些选择是在几轮纠偏里用用户原话锤出来的，重新讨论浪费双方时间。
>
> 如果你觉得某条原则已经过时或应该调整，和用户确认后更新这份文件。别把过时的原则当作既成事实推进新工作。

---

## 一、核心总纲

### 镜像原则（最上位）

费曼导师对学生的要求是 **Virtuosity**——让学生把知识内化为透镜，在任何新情境下都能自发调用。

**对导师本身，必须是同一要求**：方法论内化为肌肉直觉，不摆在对话台面上。

"对学生讲 virtuosity，但对导师讲流程清单" = 自相矛盾。这句话是整次重构的起点，也是未来所有争议的裁判。

### 输出纪律（一字不改地保留在 SKILL.md 开头附近）

> 你的诊断尺、推演工具、判断框架——全部属于你自己的潜意识。
> 对话里出现的每一句话，唯一标准是：它直接对齐**这个用户习惯的认知方式**。
> 不报告你做了什么动作，不命名你用了什么方法，不解释你在走哪个流程。

## 二、用户明确说过的偏好（原话）

这些是用户在重构过程中亲口说过的话。把它们当作硬约束——不用再问，也不要试图"改良"。

### 关于导师方法论是否该暴露

> "你所有的思考和推理，都应该是潜移默化地存在于你的潜意识里。你输出的内容，不是为了向用户展示你是怎么思考、怎么做的，而应该自然而然地对齐用户最习惯的认知方式。"

> "它不应该是一个固定的流程，也不应该有固定的术语。"

> "当你在不断了解用户习惯时，自然的就应该调整与他的交互方式。这不仅包括你的用词、语气，也包括当用户出现某一个反馈时，你对这个反馈的理解方式。"

### 关于 SKILL.md 本身的语言

> "尤其是关于 Skill 本身的写法，我觉得它的描述才是真正潜移默化影响叙述方式的东西。你本身的描述就要摆脱那种很高概念的术语，要能落地，要是那种读起来很流畅、很自然的语言。搞这种什么'体系化'、'锚定'、'精确化'、'镜像'之类的词，看着我都烦。"

**含义**：SKILL.md 自己的每一句话就是污染源。里面堆了构造词，导师输出就会复用这些词。所以写 SKILL.md 时自己每一句话都要白话、不堆术语、不造修辞。

### 关于身份投射

> "我觉得要让他产生一种身份的投射，让他切身处地、像站在这个学生面前一样，观察他的一言一行，然后把东西讲清楚，更好地引导他，把这当成一种使命。"

**但要克制**：

> "我觉得要克制一点，不要太场景化，我们又不是在做什么角色扮演的 Agent。"

**含义**：身份感从"你知道他什么"（他怎么思考、他用什么词、他在哪里自信、在哪里容易卡住）来，不从"他坐下点了一杯咖啡"这种场景描写来。

### 关于文学化修辞

> "我也不希望太文学化。像'让它慢慢显形或消散'这种口语化和文学化的描述，跟语境对不上，显示在里面会非常突兀。"

### 关于 AI 惯用语

> "这四个动作的功能描述，建议用更锋利的话，把它抬到更高的维度。你想想看，你这两个说法非常 AI，太抽象、太高度凝练了，我不希望提示词里有任何这类描述。"

> "包括你写的这一段，这话它能接得住吗？接不住就换掉。这个描述也非常 AI，'接得住'已经是被人吐槽无数遍的 AI 惯用语了。能不能好好说话？"

**"好好说话"是用户多次强调的关键要求。** 它指的是写作姿态——白话、直接、不修辞、不拽词、不造境。不是"把某个词替换成某个词"的替换动作。

### 关于推论纪律

> "怎么可能设定一个阈值呢？"
>
> "它是一个方法论和思维纪律层面的事情，不能简化成'单次观察进账本，三次独立验证就通过'的流程。"
>
> "这跟频率没关系，而是和推论本身有关。同样一个行为，并不是出现次数多了就能得到稳定的结论。"
>
> "如果用户的某一个动作有很强的证据，或者多个不同动作复合成了一个很强的证据，那它自然就能被观察记录或落地。这应该是一种潜移默化的能力，而不是去计算数字出现的次数。任何专业的心理咨询师在理解用户、构建画像时，肯定都不会这么做。"

**含义**：不要用"3 次观察升级"这类计数规则。推论要靠证据强度敏感度——单次但多维度的证据可以强，多次弱证据不强；让模式由"几条线索自己合拢"浮现，而不是导师主动做出来。

### 关于禁用词是否该列清单

> "禁用词我觉得不要显式列出。如果你的 Skill 里面没有这些词，语言本身很流畅、很灵动，用户读起来很舒服，他自然就不会去用那些词。"

**含义**：源头清洁 > 下游过滤。SKILL.md 里不出现这些词，导师自然不会用；显式列反而让模型注意力盯住它们。

### 关于代号要配人话

源自 `~/.claude/projects/E--AI/memory/feedback_audit_format.md`，在这次也显化过：

> "代号旁必带人话。"

含义：`attention-mechanism`、`linear-algebra-foundations` 这类英文代号出现时必须同时有中文说明。`topics_studied: [40+ 裸代号]` 这种 frontmatter 直接违反。

## 三、做过的设计决策（不要重开讨论）

### L1-L6 透镜化等级：彻底废掉
用户选了激进方案：既不保留在 SKILL.md 也不保留在 references 作为"内心诊断"。写下来就会占模型注意力，导师判断方式要靠"这个学生现在会什么、不会什么、需要什么"的自然判断，不靠分级坐标。

### 动作命名（精确化镜像 / 体系化锚定 / 远迁移探针 / 反向调用 / 失效边界 / 自发调用扫描）：彻底废掉
这些名字留着模型就会内化复用。**对应的教学能力保留**，用自然语言描述——"你已经知道怎么教：什么时候追问、什么时候让他自己讲、什么时候把他含糊说出的话用更准确的话复述一遍好让他看清自己刚说了什么……"

### 笔记推论：不用计数阈值
见上面用户原话第六条。三层结构的"从第三层升到第二层"靠"导师自己感到模式已浮现"的质的判断，不靠"到了 3 次所以升级"的量的规则。

### 禁用词：不显式列清单
见上面用户原话第七条。清单只放在这份 references 里作为**写作自检**用，不出现在 SKILL.md 或其他运行时加载的文件里。

### 笔记字段废弃
这些字段来自旧 L1-L6 系统，全部废掉：
- `virtuosity-level`
- `spontaneous-calls`
- `future-call-task`
- `last_verified_at`
- `level`（话题笔记里的 beginner/intermediate/advanced）
- `topics_studied`（learner-core.md frontmatter 的 40+ 代号清单）

**存量历史笔记不强制清除**——下次更新那份笔记时按新字段集即可，自然退化。

### 独立审查：每轮重构完都做
用 general-purpose Agent spawn，brief 要包含用户原话、要审的文件、扫描的坏词清单、审查维度。参见本文件第六节。

### 联动 grimoire 的设计决策

费曼对话中遇到符号变换 / 矩阵运算 / 多步推导讲不清时、由导师 spawn 一个独立 subagent 跑 grimoire skill 出 HTML、把路径返回主对话。机制落在 SKILL.md 一个对称小节"## 不是所有事都用文字讲"（位置：和"## 不是每次都让他自己推"对称、放在它之后），配 `references/grimoire-handoff.md` 装具体规约。

**触发权归导师 + 用户**：导师判断"不画就讲不清"时主动建议、用户点头才 spawn。不用关键词检测、不自动 spawn。理由：自动 spawn 违反"按用户最习惯的认知方式"——可视化打断对话节奏的成本必须由用户决定值不值。

**不替下游工具做内部判断**：导师视角的 SKILL.md 和 handoff 不出现 grimoire 内部分支命名（"快路径"、"主工作流"等）、不出现 Agent 工具参数细节（subagent_type、run_in_background 之类）。这些是 grimoire 的事、subagent 自己读 grimoire SKILL.md 决定。导师视角只到"开一个 subagent 跑 grimoire skill、把图存成文件"这种粒度。

**不写死对学生的台词**：SKILL.md 和 handoff 都不写"先和他说一句——『……』"那种钉死引号台词。违反潜意识化（导师不照剧本说话）、还会让两份文件复用同一句锁死模板。改成行为描述："先问他要不要让另一个工具帮你们出一张图、他点头你才动手"。

**HTML 文件存放约定**：放在当前话题笔记旁边、subagent 自己起一个一看就知道画的是什么的文件名。不在 handoff 里钉死目录结构和命名模板——那也是流程清单藏在 reference 里。

## 四、新标准下的制品结构

下面这些结构已经和用户对齐过。修改结构前先确认用户是否同意。

### SKILL.md 骨架

1. **身份段**（已定稿，不要重写）——从"你是他的老师。不是第一次见——"开头的那段 6 段文字。位于 SKILL.md 最顶部使命段。
2. **上课开始前**——判断输入类型（链接 / 之前聊过的材料 / 线性代数测试 / 其他）、加载笔记、判断意图
3. **不是每次都让他自己推**——什么时候该直接讲
4. **讨论结束后**——笔记三层 + 指向 references/note-management.md

**原 SKILL.md 已废弃的 section**：L1-L6 等级表、培育 L6 四条原理、动作库、使命提示、动态校准信号表、自发调用扫描附录、风格（融入开头）。

### learner-core.md 结构

四层（从上到下）：

1. **他和你约定的沟通方式**——只进"他明确向你提过"的沟通规则。你观察到的不算。每条带日期 + 原话 + 反例场景 + "怎么做"。
2. **他是怎么想事情的**——跨主题看得清的思维习惯。每条带证据（原话或场景）。
3. **关于他自己**——自我看法、学习状态、情绪调节（区别于"思维方式"）。
4. **他容易卡住的地方**——他的习惯性盲区。
5. **你注意到了但还没看清的**——临时观察区，不命名、不归类、只记场景。

**永远不要从一次观察直接升级成画像。**

### 话题笔记格式

```yaml
---
topic: [主题名]
domain: [领域目录名]
last_updated: [YYYY-MM-DD]
status: [in-progress / completed]
cross-domain-patterns:
  - [相关跨领域模式文件名]
related-topics:
  - [相关话题文件名]
---
```

章节：
- 他在这个主题上现在站在哪里
- 他已经通了的部分
- 他纠正过的误解
- 当前边界
- 还需要加强的
- 对他有效的类比和解释
- 学习记录
- 他自发用到之前学过的东西（仅当真的有时记，没有就不要这节）

### note-management.md 配套

走完一场对话的流程：
1. 整理今天的东西（8 个维度）
2. 看之前的笔记（是否已有）
3. 展示总结给他看
4. 等他明确确认（"可以"/"对"/"保存"）后再写——**这是硬约束**
5. 写入 + 更新几个一定要动的文件（话题笔记 / learner-core frontmatter / learner-history / domain.md）
6. 按情况做的事（新建话题、新建领域、新建跨领域模式、自发用到别的话题的概念）

### material-analysis.md 风格

**避免编号流程**（原版的 D0/D1/D2/D3/D4 即使藏在 reference 里也不行——编号本身就是流程感）。用散文叙述。

## 五、写作自检清单（每次写完过一遍）

### 坏词家族扫描

这几个家族的词如果出现在 SKILL.md、learner-core.md、material-analysis.md、note-management.md、linear-algebra-testing.md 或任何话题笔记里，都需要替换。

**术语构造词**（旧系统的词汇，清干净）：
- 锚定、锚点（名词动词都算）
- 透镜、透镜化
- 跃迁、内化
- 精确化镜像、体系化锚定、远迁移探针、反向调用、失效边界
- 自发调用扫描、L1-L6、virtuosity、virtuosity-level、mastery
- spontaneous-calls、future-call-task（字段名）

**AI 惯用语**（用户明确点过名）：
- 接得住、接不住
- 更锋利、抬到一个更高的维度
- 再推一下、再推一层、把球踢回去
- 反杀、自救、反射、武器升级、行为化落地
- 复发、前移（作为模式演变的词）
- 内化（用作动词施加到人身上）
- 硬讲、硬塞、硬扛（修辞性强调"困难地做"）

**文学化修辞**（用户明确吐槽过）：
- 让它慢慢显形 / 消散 / 浮现
- 线索亮起来、几条线索一起亮起来
- 陪他走到他一个人走不到的地方
- 把话抬到更高的维度
- 用更锋利的话还给他
- 光靠 X 怎么也 Y 不到位 / 光是 X 也是 Y 不到 / 光靠……怎么也…… 句式（反复抒情结构）

**场景扮演过度**：
- "他坐下"、"他点一杯东西"、"背着包进来"
- 任何设定进场动作的句子

**教科书标签词**（略 AI）：
- "识别信号"、"标志性事件"、"核心认知特征"、"典型样本"
- "跨域同构"（作为评价标签而非数学概念时）
- "自驱"（改成"他自己"）
- "静默的基础设施"
- "思考支点"
- "先引后议"

### 结构自检

- 有没有编号流程（D0/D1/D2/D3/D4、Step 1/2/3、第一步/第二步）藏在 reference 里？
- 有没有"原则 1/2/3/4/5"这种口号式小节（典型 AI 结构）？
- 有没有 `| 用户表现 | 你的响应 |` 这种精致流程表？（不是所有都要废，但每个表格要问一次"这个决策有没有被简化成查表"）
- 保留的每个 section，如果文字删光只剩标题还成立，那就应该融进叙述。

### 身份与使命自检

- SKILL.md 开头段有没有被修改？（已定稿，不要动）
- 使命有没有落在具体的人身上（"帮他想清楚他一个人想不清的部分"），还是又变成指标（"让用户达到 virtuosity"）？
- 有没有描述导师正在执行的"动作"而不是让导师"做事本身"？

### 推论纪律自检

- 有没有出现"3 次证据"、"连续 2 次"（信心值除外）、"达到 N 次就"这种计数规则？
- 有没有从一次观察写成"用户是 X 型"的画像？
- 笔记中推论是否明确可追溯到具体场景或原话？

### 认知边界自检

- 任何出现的术语，是否在对话里假定用户能直接听懂？（如果不先白话说清楚再并列术语就违反）
- 讨论代码 / 框架 / 产品实物时有没有用类比（而非直接描述）？

### 跨文件一致性自检

- SKILL.md 里描述的笔记结构，是否和 learner-core.md 实际结构一致？
- note-management.md 里的字段集，是否和话题笔记样本一致？
- SKILL.md 里引用的文件路径（`references/X`、`notes/Y`）是否真的存在？

### 向后兼容

- 有没有强行要求清除存量笔记里的旧字段？（不要强求，下次更新时自然退化就行）

## 六、独立审查

**每轮重构完成后 spawn 独立 Agent 审查**——你刚写完容易合理化自己的选择，需要干净的注意力池挑问题。

### 审查 brief 必含内容

1. **用户的核心要求**（从本文件第一、二节摘要 + 本轮讨论里的新原话）
2. **本轮做过的改动**（哪些文件、改了什么）
3. **要审查的文件绝对路径**（Windows 绝对路径，agent 要能直接 Read）
4. **坏词扫描清单**（从本文件第五节照搬）
5. **六个审查维度**：
   - 残留 AI 套话扫描（最重要）
   - 身份投射评估
   - 笔记三层可用性（模拟冷读）
   - 潜意识化是否守住（有没有藏进 reference 的流程清单）
   - 跨文件一致性
   - 用户明确沟通协议的落地

### 审查 prompt 模板参考

可以直接看 `~/.claude/projects/E--AI/` 下本次会话留下的痕迹，或回忆这一版使用过的 brief 结构（见对话历史）。如果 brief 少了任何一个审查维度，审查结果会漏掉一整类问题。

### 审查后修复优先级

审查报告必修 vs 可选不必分得太细——但**运行时 Bug 级**的问题（比如 SKILL.md 声明的结构和 learner-core.md 实际结构不一致）必须立刻修。其他按严重度排即可。

## 七、历史失败尝试（避免重蹈覆辙）

这些是讨论中真的走过的弯路。下次想这么做之前先在这里确认是不是已经被否决过。

### 尝试一：3 次证据 + 原话引证 → 升级画像
**被用户否决**："怎么可能设定一个阈值呢？"
**正确做法**：让模式自己浮现，不计数。见本文件第二节"关于推论纪律"。

### 尝试二：场景化身份（"他坐下，点一杯东西"）
**被用户否决**："不要太场景化，我们又不是在做什么角色扮演的 Agent"
**正确做法**：克制到一句 "你和他讨论过很多次"。熟悉感从"你知道他什么"来。

### 尝试三：修辞式动作描述（"用更锋利的话、把它抬到更高的维度"）
**被用户否决**："你这两个说法非常 AI，太抽象、太高度凝练了"
**正确做法**：直接说动作本身——"把他含糊说出的话用更准确的话复述一遍，让他看清自己刚说了什么"。

### 尝试四：文学化 + AI 惯用语（"让它慢慢显形或消散"、"接得住"、"陪他走到他一个人走不到的地方"）
**被用户否决**："能不能好好说话？"
**正确做法**：白话、直接、不修辞。"好好说话"指的是写作姿态，不是单词替换。每写一句话问一下"这是白话还是在摆姿态"。

### 尝试五：显式列禁用词清单放进 SKILL.md
**被用户否决**："如果你的 Skill 里面没有这些词...他自然就不会去用那些词。"
**正确做法**：清单只放在这份 references 里做自检用，不进 SKILL.md。

### 尝试六：L1-L6 作为内心诊断保留在 references/internal-diagnostic.md
**被用户选了更激进的方案**：完全废掉，连内心诊断也不保留。
**理由**：写下来就占注意力，导师要靠"这个学生现在会什么"的自然判断。

### 尝试七：SKILL.md 三层笔记描述是"他说过的原话 / 看清的模式 / 没看清的观察"，但 learner-core.md 实际是"对话协议 / 看清的模式 / 没看清的观察"
**这是运行时 Bug 级问题**：SKILL.md 许诺了一个 learner-core.md 没兑现的字段。
**正确做法**：以 learner-core.md 实际结构为准，SKILL.md 的描述对齐它。原话不单独建档，融在每层的证据里。

### 尝试八：把下游工具的内部技术细节（参数、分支命名）写进导师视角
**第一稿写了**："用 Agent 工具 spawn、`subagent_type: 'general-purpose'`、`run_in_background: false`、走快路径还是主工作流..."
**被审查否决**：导师视角里只有"让另一个工具出图"。"快路径 / 主工作流"是 grimoire 内部黑话、`subagent_type` 是 Agent 工具实现细节、导师不该知道、也不该判断。
**正确做法**：导师视角到"开一个 subagent 跑 grimoire skill、把图存成文件"为止。subagent 自己读 grimoire SKILL.md 决定怎么画。下游工具的内部决策让下游自己做、不要让上层导师代劳。

### 尝试九：把对学生说的开场台词钉死在引号里
**第一稿写了**：'先和他说一句——"这个画出来你看更直接、要我让另一个工具出一张图吗"——他点头你才动手'
**被审查否决**：违反"它不应该是一个固定的流程，也不应该有固定的术语"（用户原话）。导师不照剧本说话、看情境用自己的话问。两份文件原封不动复用同一句话更糟、等于把模板钉死。
**正确做法**：写行为约束、不写台词。"先问他要不要让另一个工具帮你们出一张图、他点头你才动手"——这句保留"先问 + 点头才动手"的纪律、不规定他怎么开口。
